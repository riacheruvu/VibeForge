SELF-IMPROVEMENT LOOP

Improve one part of the setup at a time

Use real friction to create tests, compare one focused change, and keep it only when the same checks improve.

  1. Find frictionImport conversations or add a public-safe example.
  2. Make testsApprove only rewrites that capture the actual preference.
  3. Change one thingTry a model, prompt, memory, skill, tool, or routing change.
  4. Gate the resultUse final-check cases before treating the setup as better.
Keep the loop humble A better score is a reason to inspect the outputs, not permission to deploy a change automatically.
What VibeCheckBench can do todayIt can directly compare models and system prompts. It can also track planned changes to memory, skills, tools, context, and routing while their execution adapters are added.
